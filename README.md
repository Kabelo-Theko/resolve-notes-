# resolve-notes

**Live demo:** _(set NVIDIA_API_KEY in Vercel, then add the URL here)_

Turn a resolved ticket into a clean knowledge base article. Give it the issue,
what you tried and what fixed it, and it writes a consistent KB entry: title,
symptoms, root cause, numbered fix steps, a prevention note and tags. You can
copy it as Markdown, download it, or save it to a searchable library. Every AI
draft carries a short reasoning receipt (which model, what it read, when) and is
saved as *needs review* until a person approves it.

This is the last step of the support loop the other tools cover:

- [onboard-kit](https://github.com/Kabelo-Theko/onboard-kit): a new starter arrives
- [ticket-triage](https://github.com/Kabelo-Theko/ticket-triage): they report a problem
- [net-doctor](https://github.com/Kabelo-Theko/net-doctor): a fault is diagnosed
- **resolve-notes**: the fix is written up so the next person does not start from scratch

## How it works

The article *structure* is enforced by the app; the *wording* is generated by an
LLM through a small serverless function (`api/ai.js`). The API key lives only in
the `NVIDIA_API_KEY` environment variable in the Vercel project, never in the
repo or the browser. Without the key the app still runs: the "See an example"
button shows the output format, and the rule of the tool (the fixed structure)
is unchanged.

The saved library uses the browser's `localStorage` in this demo. A production
build would persist articles to a shared database (Supabase) so the whole team
sees the same knowledge base.

## In the knowledge base

- **Reasoning receipt.** Every AI-drafted article shows which model wrote it, what it read (your fields or a pasted thread) and when. The structure is the app's; only the wording is the model's.
- **Review before trust.** AI drafts save as *needs review* and stay that way until someone clicks **Approve**, so nothing reaches the team as fact unchecked.
- **Search, filter, related.** The library has full-text search and one-click tag filters, and an open article lists related notes that share its tags.
- **Export.** Download the whole library as JSON for backup or to hand to another tool.

## Connecting it to the other tools

resolve-notes is the knowledge spine of the set, and it links by plain URL with
no backend: `?view=library`, `?q=printer` (pre-searched) or `?tag=gateway`
(pre-filtered), so net-doctor can deep-link straight to the history for a
symptom. A same-page embed can also call `window.resolveNotes.search("gateway")`.
Review status and the reasoning receipt are kept in the browser; Supabase, when
enabled, stores the core article.

## Run / deploy

It is a static front end in `docs/` plus one serverless function in `api/`.

- **Vercel:** import the repo, add `NVIDIA_API_KEY` in Settings → Environment
  Variables, deploy. `vercel.json` serves `docs/` and runs `api/ai.js`.
- **Static only (no AI):** open `docs/index.html`; the example and format work
  without a backend.

## Choosing the AI model

The Generate panel has a model selector: **DeepSeek v4 Flash** (fast, the
default), **DeepSeek v4 Pro** (best quality, slower), or **MiniMax M3**. The
serverless function routes to the chosen model. One `NVIDIA_API_KEY` powers all
three; you can also set `NVIDIA_API_KEY_PRO` / `NVIDIA_API_KEY_MINIMAX` to use
separate keys per model. (Pro is slower, so the function allows up to 60s.)

## Enabling shared persistence (Supabase, optional)

By default the library is stored in the browser. To share a knowledge base
across a team, set `SUPABASE_URL` and `SUPABASE_KEY` (a **publishable** key, safe
in the client) near the top of `docs/index.html`, and create the table:

```sql
create table if not exists kb_articles (
  id text primary key,
  title text, symptoms text, root_cause text,
  fix_steps jsonb, prevention text, tags jsonb,
  created_at timestamptz default now()
);
alter table kb_articles enable row level security;
create policy "demo read"   on kb_articles for select using (true);
create policy "demo insert" on kb_articles for insert with check (true);
create policy "demo delete" on kb_articles for delete using (true);
```

Row-level security controls access; the demo policies above are open for a
public demo, so tighten them for real use.
